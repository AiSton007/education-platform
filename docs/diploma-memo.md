## Слайд 3 — Инфраструктура K8s-кластера

### Что показано
Топология кластера: Windows-хост, DNS/NFS-нода, control1 и три worker-ноды. Таблица всех компонентов платформы.

### Что говорить
> «Кластер развёрнут на виртуальных машинах VirtualBox в локальной сети 192.168.1.0/24.
> Отдельная нода выполняет роль DNS-сервера (bind9), NFS-хранилища для persistent volume и Ansible-контроллера для управления.
> Control-plane нода управляет кластером. Три worker-ноды принимают нагрузку.
> Сетевой плагин Calico с режимом IPIPCrossSubnet обеспечивает связность подов и сетевые политики.
> MetalLB выдаёт LoadBalancer-адреса для ingress-nginx, cert-manager управляет TLS-сертификатами через внутренний CA.»

### Возможные вопросы
- **Почему self-hosted, а не облако?** — Полный контроль над инфраструктурой, возможность изучить компоненты изнутри.
- **Что такое NodeLocalDNS?** — Кэширует DNS-запросы на каждой ноде, снижает нагрузку на CoreDNS и latency.
- **Что такое nfs-subdir?** — Provisioner для динамического создания PersistentVolume из NFS-директорий.

## Слайд 4 — Архитектура приложения

### Что показано
Контекстная диаграмма: три типа пользователей → Frontend SPA → API Gateway → 5 backend-сервисов. Принципы монорепозитория.

### Что говорить
> «Архитектурно система следует паттерну API Gateway: единственный публичный endpoint для frontend — это api-gateway, который проверяет JWT и проксирует запросы к внутренним сервисам.
> Все сервисы расположены в одном монорепозитории с единым pyproject.toml и пакетным менеджером uv. Зависимости каждого сервиса объявлены как optional-dependencies extras.
> Общий код: конфигурация, логирование, JWT-авторизация, работа с БД, health и metrics — вынесен в пакет pkg/ и переиспользуется всеми сервисами.
> Следуем 12-factor app: никаких hardcoded IP, все секреты через переменные окружения.»

### Возможные вопросы
- **Что такое API Gateway?** — Proxy-сервис, который стоит перед всеми backend-сервисами. Проверяет JWT, добавляет request_id, возвращает единый формат ошибок.
- **Зачем монорепозиторий?** — Единый стандарт кода, общие зависимости, проще запустить локально и тестировать.

## Слайд 5 — Сервисы платформы

### Что показано
Таблица всех 7 сервисов с портами, схемами БД и ответственностью. Карточки: healthcheck endpoints, internal JWT, миграции.

### Что говорить
> «Каждый сервис — это независимый FastAPI-процесс со своей схемой в PostgreSQL.
> auth-service отвечает только за токены — он не хранит профиль, а обращается за данными пользователя в user-service через внутренний JWT.
> test-service — ключевой оркестратор: именно он управляет полным бизнес-процессом прохождения теста: сохраняет ответы, вызывает llm-service и report-service.
> Каждый сервис реализует три обязательных эндпоинта: /healthz для liveness probe, /readyz для readiness с проверкой БД, /metrics для Prometheus.
> Межсервисные вызовы защищены отдельным Internal JWT с заголовком X-Internal-Token и TTL 5 минут.»

### Возможные вопросы
- **Зачем отдельный internal JWT?** — Чтобы внутренние эндпоинты не могли вызвать внешние пользователи, даже имея валидный access token.
- **Почему не gRPC?** — HTTP достаточен для данной нагрузки, проще дебажить и документировать через Swagger.

### Возможные вопросы

- **Что такое uv?** — Rust-реализация менеджера Python-пакетов. Уменьшает время установки зависимостей в 10–100 раз.

## Слайд 7 — Поток: Аутентификация

### Что показано
Sequence-диаграмма: User → Frontend → Gateway → auth-service → user-service → DB для login и /me.

### Что говорить
> «Рассмотрим поток аутентификации. Пользователь вводит логин и пароль, frontend отправляет запрос через Gateway в auth-service.
> auth-service получает из БД запись пользователя, проверяет bcrypt-хэш пароля и выпускает access + refresh JWT.
> При запросе GET /auth/me auth-service получает user_id из JWT и обращается в user-service за профилем через внутренний токен.
> Таким образом auth-service не хранит профильные данные — это ответственность user-service. Разделение обязанностей строго соблюдается.»

### Возможные вопросы
- **Где хранится refresh token?** — В таблице sessions схемы auth в PostgreSQL. При logout — удаляется.
- **Что в payload JWT?** — sub (user_id), role, iss, aud, exp. Роль позволяет не обращаться в БД при каждой авторизации.

## Слайд 8 — Поток: Прохождение теста

### Что показано
Sequence-диаграмма: Employee → Frontend → Gateway → test-service → llm-service → report-service → DB.

### Что говорить
> «Ключевой бизнес-поток — прохождение теста. Сотрудник отправляет ответы через frontend.
> test-service принимает их, сохраняет в БД, затем последовательно вызывает llm-service для анализа и report-service для формирования отчёта.
> Именно test-service является оркестратором: он знает об обоих сервисах. llm-service и report-service изолированы — они не знают друг о друге.

### Возможные вопросы
- **Что происходит если LLM недоступен?** — test-service получает ошибку, попытка сохраняется в статусе failed. В dev-окружении используется mock-провайдер.
- **Синхронный или асинхронный поток?** — Синхронный HTTP. В будущем можно заменить на очередь (NATS/Kafka) для фонового анализа.

## Слайд 9 — Модель данных

### Что показано
ER-диаграмма ключевых сущностей схемы tests. Таблица всех 5 схем. Callout о решении Enum-проблемы.

### Что говорить
> «Используется один инстанс PostgreSQL, разделённый на пять изолированных схем по числу сервисов с БД.
> Центральная схема — tests. Тест содержит вопросы, назначения связывают тест с пользователем, попытки — это конкретные прохождения с ответами. Отчёт привязан к попытке.
> Важное техническое решение: SQLAlchemy по умолчанию сохраняет в БД имя enum-константы (FREE_TEXT), тогда как PostgreSQL ожидает значение (free_text). Это несоответствие привело к ошибкам в продакшене. Исправлено через values_callable и validate_strings во всех моделях.»

### Возможные вопросы
- **Почему одна база, а не разные?** — Для MVP достаточно одной БД с изоляцией по схемам. Физическое разделение — следующий шаг масштабирования.
- **Как применяются миграции в K8s?** — Kubernetes Job, запускаемый как PreSync hook ArgoCD. Сервис стартует только после успешного завершения Job.

## Слайд 11 — CI/CD пайплайн

### Что показано
Схема пайплайна из 9 этапов (git push → deploy). Три карточки: Kaniko, ArgoCD, оптимизации.

### Что говорить
> «Весь путь от git push до продакшена автоматизирован.
> Jenkins запускает пайплайн: устанавливает зависимости через uv, прогоняет ruff-lint и pytest.
> Затем Kaniko собирает 12 Docker-образов без Docker daemon — это важно для Kubernetes, где privileged mode нежелателен.
> Образы пушатся в Harbor с тегами SHA коммита и latest. Jenkins обновляет Helm values, делает git push — и ArgoCD автоматически применяет изменения.
> Порядок применения управляется sync-waves: сначала PostgreSQL, затем migrate Jobs, потом сами Deployments.»

### Возможные вопросы
- **Зачем Kaniko, а не docker build?** — Docker требует privileged-контейнер в K8s. Kaniko выполняет сборку в user space без привилегий.
- **Что такое GitOps?** — Git — единственный источник истины для состояния кластера. ArgoCD сверяет желаемое (Git) с фактическим (K8s) и синхронизирует.
- **Почему два тега (SHA + latest)?** — SHA для детерминированного rollback, latest для удобства при ручных проверках.

## Слайд 13 — Инженерные решения

### Что показано
4 карточки: Enum mismatch, MissingGreenlet, PDF кириллица, pycairo в Kaniko — каждая с симптомом и решением.

### Что говорить
> «В ходе разработки было выявлено и решено несколько нетривиальных проблем.
> Первая — несоответствие Enum между SQLAlchemy и PostgreSQL: ORM сохранял имя константы, база ожидала значение. Исправлено принудительным маппингом через values_callable.
> Вторая — ошибка MissingGreenlet в async SQLAlchemy: lazy-load отношения вне сессии. Решение — вычислять count через scalar_subquery прямо в основном запросе.
> Третья — кириллица в PDF: xhtml2pdf игнорировал CSS @font-face. Решение — явная регистрация шрифта DejaVuSans в ReportLab перед генерацией.
> Четвёртая — системные зависимости cairo/pango для сборки pycairo внутри Kaniko. Решение — базовый Python-образ с предустановленными системными пакетами.»

### Возможные вопросы
- **Почему это важно?** — Это реальные production-инциденты, не синтетические задачи. Демонстрирует практическое владение стеком.
- **Что такое greenlet?** — Механизм корутин для async-совместимости SQLAlchemy. При lazy-load вне async-контекста падает с этой ошибкой.

## Частые вопросы на защите (вне слайдов)

### Общие
- **Какую проблему решает платформа?** — Автоматизация корпоративного тестирования с LLM-анализом и генерацией рекомендаций.
- **Сколько времени заняла разработка?** — Отвечай честно по факту.
- **Что самое сложное?** — Настройка GitOps-цепочки и устранение производственных ошибок (enum, greenlet, PDF).

### По архитектуре
- **Чем микросервисы лучше монолита?** — Независимый деплой, масштабирование, изоляция сбоев, возможность разных технологий.
- **Какой сервис наиболее нагруженный?** — test-service, он оркестрирует все запросы прохождения теста.
- **Как сервисы находят друг друга?** — Через Kubernetes DNS: `http://user-service:8080` — это CoreDNS резолвит в ClusterIP сервиса.

### По безопасности
- **Как защищены пароли?** — bcrypt с автоматической солью через passlib.
- **Что будет если захватить один pod?** — NetworkPolicy ограничивает его возможности до минимума. ReadOnly FS не позволяет записать код.

### По CI/CD
- **Что произойдёт если тесты упадут?** — Jenkins прервёт пайплайн, образы не соберутся, в кластер ничего не попадёт.
- **Как проверить что деплой прошёл успешно?** — ArgoCD показывает статус синхронизации, readiness probe на podах гарантирует готовность.
